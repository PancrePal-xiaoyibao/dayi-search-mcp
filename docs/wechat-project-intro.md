# 我们做了一个「医药信息检索助手」：从关键词到结构化结果，一键可用

在医疗内容场景里，很多人都有同一个痛点：

- 想查一个药，比如“替吉奥”“吉西他滨”，结果网页信息很长、很散；
- 想查医生，比如“傅德良”，常常要在多个页面来回点；
- 想把查到的信息交给 AI 做进一步总结，却发现数据不够结构化，难以复用。

所以我们做了一个项目：**dayi-search-mcp**。  
它的目标很简单：**把“网页检索”变成“结构化可用数据”**，并直接接入 AI 工具链。

---

## 这个项目能做什么？

### 1）统一检索 4 类医学信息

支持按关键词检索：

- 药品（medical）
- 疾病（disease）
- 医生（doctor）
- 症状（symptom）

比如输入“白蛋白紫杉醇”“傅德良”“吉西他滨副作用”，都能走同一套查询流程。

---

### 2）提取更完整、更可读的药品信息

针对药品，项目会重点输出结构化字段，包括：

**药品详情**
- 成分
- 性状
- 适应症
- 用法用量
- 规格
- 贮藏方法
- 有效期
- 执行标准

**注意事项**
- 不良反应
- 禁忌
- 药物相互作用
- 注意事项

这意味着：不仅能“查到”，还可以“直接用于分析和总结”。

---

### 3）支持原文证据保留，适合 RAG

除了结构化字段，项目还保留原始证据：

- detail_html
- nuxt_script
- search_payload

这让它非常适合后续做：
- 医疗知识库
- RAG 检索增强
- 审计与溯源

---

### 4）查询结果自动落盘，方便复用

每次查询可以直接保存为本地 JSON，默认会写入：

`/tmp/标题_YYYYMMDD.json`

例如：
- `/tmp/傅德良_20260407.json`
- `/tmp/吉西他滨_20260407.json`

后续导入向量库、工作流系统、LLM agent 都很方便。

---

## 为什么它和“普通网页搜索”不一样？

核心差异在于：**它不是只返回一段文本，而是返回“可计算、可追溯、可复用”的数据结构。**

简单说：

- 传统搜索：给你网页
- 这个项目：给你**结构化结果 + 原文证据 + 可直接接 AI 的接口**

---

## 可接入哪些 AI 平台？

项目已经提供 MCP 能力，可对接：

- Cherry Studio
- FastGPT
- OpenClaw
- Claude Code
- Codex

并支持标准 stdio 方式接入，适合本地和工作流环境。

---

## 这件事的价值是什么？

我们认为，医疗信息工具的下一步不只是“搜到内容”，而是：

1. **可理解**（结构化字段清晰）
2. **可验证**（保留原文证据）
3. **可扩展**（MCP 接入多客户端）
4. **可自动化**（可落盘、可接 RAG、可接 Agent）

这也是我们做这个项目的初衷：  
让医学信息从“网页内容”升级为“AI 时代的可用数据资产”。
